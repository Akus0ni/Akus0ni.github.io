Angular 19 Internals
01

Bootstrap & the standalone application

What happens in the milliseconds between index.html loading and the first component painting to the screen.

What we are decompiling

This course reads one real codebase — a single-page developer portfolio built on Angular 19. There is no backend and no runtime data fetching: every piece of content is static TypeScript, rendered by Angular using signals and scroll-driven UI. That makes it an ideal specimen — nothing is hidden behind a network call, so the entire machine is the framework itself.

We are only teaching the Angular layer, at the level of an engineer who wants to know why the framework behaves the way it does, not just how to call its APIs.

🧭
The mental model to hold

An Angular app is a tree of components, driven by a change detection loop, wired together by dependency injection, and increasingly steered by signals. Those four systems are the whole course. This module starts the tree.

The entry point is six lines

A browser cannot run an Angular component directly. Something has to create the root component, attach it to the DOM, and start the runtime. That bootstrap ritual lives in src/main.ts.

main.ts 
import { bootstrapApplication } from '@angular/platform-browser';
import { appConfig } from './app/app.config';
import { AppComponent } from './app/app.component';

bootstrapApplication(AppComponent, appConfig)
  .catch((err) => console.error(err));
What Angular does

Pull in the standalone bootstrap function — the modern replacement for the old platformBrowserDynamic().bootstrapModule() path.

Import the application config and the root component class.

Create the root environment injector from appConfig, instantiate AppComponent, and render it into the element matching its selector (app-root in index.html).

Bootstrap returns a Promise; if anything in construction throws, log it. This is the app's outermost error boundary.

💡
No NgModule in sight

Before v14, every app was rooted in an @NgModule that declared components, imported other modules, and listed providers. bootstrapApplication deletes that ceremony: the root component plus a flat providers array is the entire application definition. In Angular 19 this standalone path is the default.

"Bootstrap" really means "compile ahead of time, then run"

The six lines above are the runtime step. The heavier work already happened at build time. Angular's compiler — Ivy — reads every component's template during an AOT build and rewrites it into imperative instruction functions.

⚙️
Why this matters for you

There is no template interpreter in the browser. Your template is compiled code by the time it runs. This is why template type errors fail your build, why Angular bundles are relatively lean, and why the runtime can be so aggressive about only touching the DOM nodes that actually changed.

The application config: providers are the app's wiring

The second argument to bootstrapApplication is where you configure the root injector. This app registers exactly two things.

app.config.ts 
import { ApplicationConfig, provideZoneChangeDetection } from '@angular/core';
import { provideRouter } from '@angular/router';
import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [provideZoneChangeDetection({ eventCoalescing: true }), provideRouter(routes)]
};
What Angular does

provide* functions are the standalone idiom: each returns a set of providers to merge into the root injector.

provideZoneChangeDetection installs the Zone.js-based change detection runtime, and eventCoalescing: true tells it to batch multiple DOM events fired in one task into a single change detection pass (Module 06).

provideRouter(routes) registers the router and its route table — even though this single-page app barely uses it, it is part of the wiring.

This flat array is the app's configuration. Everything else is discovered through the component tree and providedIn: 'root' services.

provideZoneChangeDetection Installs Zone.js change detection; eventCoalescing batches events into one CD run per task.
provideRouter Registers the Router, route matching, and navigation services into the root injector.

02

The component model

A component is a class plus a decorator that tells the compiler how to turn it into a rendering unit. In v19 it stands alone — no module required.

Class + decorator = component

An Angular component is nothing more than a TypeScript class annotated with the @Component decorator. The class holds state and methods; the decorator holds the metadata the compiler needs — which HTML element to hook onto, what template to render, which other components the template is allowed to use.

🏷️
selector

The CSS selector Angular matches in host templates to instantiate this component — e.g. app-root.

📄
template / templateUrl

The view, compiled by Ivy into instruction functions. Inline for small components, external file for large ones.

📦
imports

The exact list of components, directives, and pipes this template is permitted to reference.

The root component, decoded

Here is the decorator on AppComponent — the composition root that assembles every page section.

app.component.ts 
@Component({
  selector: 'app-root',
  standalone: true,
  imports: [
    CommonModule, HeroComponent, AboutComponent, ExperienceComponent,
    ProjectsComponent, QueryConsoleComponent, SkillsComponent,
    EducationComponent, ContactComponent,
  ],
  templateUrl: './app.component.html',
  styleUrl: './app.component.scss',
})
export class AppComponent implements AfterViewInit {
What Angular does

Wherever <app-root> appears (here, in index.html), instantiate this class and render its template inside.

standalone: true marks it as needing no NgModule. In v19 this is the default — writing it is now optional but explicit.

The imports array is the template's dependency allowlist: these nine symbols are the only components/directives its template may use. CommonModule brings pipes like date and async.

The view and its scoped styles live in external files — the convention for a large component. Ivy compiles both at build time.

🔒
Imports are a hard boundary

If a template references a component that is not in its imports array, the build fails. This makes each component's dependencies local and explicit — the opposite of the old NgModule world, where a component could silently use anything the surrounding module happened to import. When you steer an AI to "add a component to this template," the imports array is what must change with it.

Composition, not inheritance

The root template does almost nothing but place the section components in order. The tree is the layout. Each section is an independent, self-contained standalone component.

🌳
Why the tree shape matters

Change detection walks this tree top-to-bottom (Module 06) and dependency injection resolves along it bottom-to-top (Module 05). The shape you compose here directly determines how those two systems behave. A flat composition root like this keeps both predictable.

Inline everything: a self-contained component

Small components put template and styles inline in the decorator. Here is the entire typewriter — notice it also opts into a different change detection strategy.

typewriter.component.ts 
@Component({
  selector: 'app-typewriter',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<span class="tw">{{ text() }}<span class="blink-caret"></span></span>`,
  styles: [`.tw { font-family: var(--font-mono); }`],
})
export class TypewriterComponent implements OnInit, OnDestroy {
  @Input({ required: true }) phrases: string[] = [];
  readonly text = signal('');
What Angular does

A tiny, focused component — template and styles are inline strings rather than files.

OnPush opts this component out of default checking; it re-renders only when its signal text changes. We unpack this in Module 06.

{{ text() }} reads a signal in the template. Because it is a signal, OnPush knows precisely when to refresh this view.

@Input({ required: true }) declares a mandatory input — the compiler errors if a parent forgets to pass phrases.

Styles are scoped by default

That .tw rule in the typewriter does not leak to the rest of the page. Angular's default Emulated view encapsulation rewrites each component's selectors and stamps its elements with a unique attribute, so component CSS behaves as if scoped — without needing native Shadow DOM.

🎭

Emulated (default)

Selectors rewritten with attribute markers. Isolation without Shadow DOM. What this app uses.

🌑

ShadowDom

Real browser Shadow DOM. True isolation, but global theming (CSS variables) must pierce the boundary.

🌍

None

Styles go global, unscoped. Used deliberately for app-wide resets in styles.scss.

🎨
How theming still works

Scoped styles isolate rules, but CSS custom properties inherit through the DOM regardless of encapsulation. That is why every component can write var(--accent) and instantly respond when the theme flips — the subject of Module 04's ThemeService.

We have the units and the tree. Now: what actually goes inside a template, and how Angular's built-in control flow renders lists and conditions.

03

Templates & built-in control flow

How binding syntax connects class state to the DOM, and how @for / @if / @switch replaced the old structural directives with compiled control flow.

Five ways to bind, one direction each

Every character of dynamism in an Angular template is one of five binding forms. Each compiles to a specific Ivy instruction; there is no general expression evaluator at runtime.

{{ }}

Interpolation

{{ expr }} — writes a string into a text node. Class → DOM.

[ ]

Property binding

[prop]="expr" — sets a DOM property or component input. Class → DOM.

( )

Event binding

(event)="handler()" — runs a method on a DOM/output event. DOM → class.

[( )]

Two-way

[(x)] — sugar for a property binding + an xChange event. Both directions.

[attr.]

Attribute / class / style

[attr.x], [class.y], [style.z] — bind to attributes, classes, styles when no DOM property exists.

🧩
Property vs attribute

[value] sets the live DOM property; [attr.aria-selected] sets the HTML attribute. Angular exposes both because ARIA state, data-*, and SVG live in attribute-space with no matching JS property. This app uses [attr.data-idx] and [attr.aria-selected] for exactly that reason.

Four binding forms in one input

The skills search box is a compact tour of the syntax — property, event, interpolation, and a two-way pattern split into its parts.

skills.component.ts — template 
<input #box class="cmd-input mono" type="text"
       [ngModel]="query()" (ngModelChange)="onQuery($event)"
       (keydown)="onKey($event)"
       placeholder="Filter {{ rows.length }} skills…" />
<span class="cmd-count mono">{{ filtered().length }}</span>
What Angular does

[ngModel]="query()" — property binding pushing the query signal's value into the input. Instead of the usual [(ngModel)] two-way sugar, it is split manually.

(ngModelChange)="onQuery($event)" — the other half: on each keystroke, call onQuery with the new value, which writes the signal. Splitting the two-way binding lets the handler also reset the selection index.

(keydown)="onKey($event)" — event binding for arrow-key navigation; $event is the raw KeyboardEvent.

{{ rows.length }} and {{ filtered().length }} — interpolation, the second reading a signal so the count re-renders as results filter.

🔁
[(x)] is just [x] + (xChange)

Two-way binding is not magic — it is a naming convention. [(ngModel)]="q" desugars to [ngModel]="q" (ngModelChange)="q = $event". Seeing it split here shows the seam: any two-way binding can be pulled apart when you need to intercept the update, exactly as this code does to reset the highlighted row.

@for and the mandatory track

Since v17, loops and conditionals are built-in blocks in the template — not directives. Here is the results list.

skills.component.ts — @for 
@for (r of filtered(); track r.name; let i = $index) {
  <li role="option" [attr.aria-selected]="i === index()">
    <div class="cmd-row" [class.sel]="i === index()"
         (mouseenter)="index.set(i)"> … </div>
  </li>
} @empty {
  <li class="cmd-none mono">// no symbols match</li>
}
What Angular does

Iterate filtered() — itself a signal, so the list re-renders whenever the filter recomputes.

track r.name is required. It gives each row a stable identity so Angular can match old DOM to new data across re-renders.

let i = $index aliases the built-in index variable; also available: $count, $first, $last, $even, $odd.

@empty renders when the collection is empty — replacing the old pattern of a sibling *ngIf guard.

Why track decides reuse vs. rebuild

When the array changes, Angular diffs the new items against the tracked keys. Matching keys → the existing DOM node and its component state are reused, only bindings update. Missing keys → nodes are destroyed; new keys → nodes are created. Step through a filter operation:

The classic track mistake

Tracking by object identity (track item) when the array is rebuilt from fresh objects — e.g. after a map or an HTTP response — makes every key look new, so Angular destroys and recreates every row on each change. That is the single most common Angular list-performance bug. Track by a stable domain id (r.name here), or by $index only for static lists.

@switch, and when not to reach for it

The third control-flow block, @switch, picks one branch by strict equality — no fall-through, no break. This codebase never uses it, so the snippet below is illustrative: imagine the SQL console wanting different markup per token kind.

illustrative — not in the repo 
@switch (token.c) {
  @case ('tok-key') { <strong class="kw">{{ token.t }}</strong> }
  @case ('tok-str') { <em class="str">{{ token.t }}</em> }
  @case ('tok-fn')  { <a class="fn">{{ token.t }}</a> }
  @default          { <span>{{ token.t }}</span> }
}
What Angular does

Evaluate token.c once, then render the single @case whose value === it.

No fall-through: exactly one branch renders, so there is no break and no accidental double-render.

@default covers unmatched values — here, plain identifiers and operators.

Each branch is a full template region, so the cases can differ in structure, not just attributes.

🎯
Why the real console does not use @switch

Every token renders the same element and only the class differs. So the console writes one line — <span [class]="t.c">{{ t.t }}</span> — a plain class binding, not seven @case branches. Rule of thumb: reach for @switch when the branches produce different markup or components; reach for [class]/[ngClass] when only styling varies. Choosing the lighter tool is the mark of reading templates well.

Blocks, not directives

Old Angular used structural directives — *ngIf, *ngFor, *ngSwitch — imported from CommonModule. The new control flow is part of the template language itself.

⚡
Nothing to import

@if/@for/@switch are compiler built-ins. No CommonModule dependency, no directive resolution step.

🏎️
Compiles to direct instructions

The block lowers straight to conditional/loop instruction calls — no ng-template indirection, less overhead than the directive it replaces.

✅
track is enforced

What used to be an optional trackBy function is now a required part of the syntax — the framework makes the fast path the default path.

Templates read state; the next question is what that state actually is. In this codebase it is signals — the reactive core.

04

Signals & the reactive graph

The reactive primitive that this whole app uses instead of RxJS subjects: a value that knows who reads it and notifies them when it changes.

A signal is a value with a memory of its readers

A signal is a zero-argument getter function wrapping a value. The trick is in the two operations:

👁️
Read — mySignal()

Returns the current value. If called inside a reactive context (a computed or effect), it registers that context as a dependent.

✍️
Write — .set(v) / .update(fn)

Replaces the value (or derives it from the previous one) and marks every dependent as stale — without re-running anything yet.

📐
The type distinction

signal(0) returns a WritableSignal. A computed() returns a read-only Signal — no .set, no .update. The compiler enforces the difference, so derived state literally cannot be mutated by accident.

The three writable signals behind the skills filter

The command-palette section holds all its UI state in three writable signals — search text, active category, highlighted row.

skills.component.ts 
readonly query = signal('');
readonly activeCat = signal<string | null>(null);
readonly index = signal(0);

setCat(group: string | null): void {
  this.activeCat.update((cur) => (cur === group ? null : group));
  this.index.set(0);
}
What Angular does

Three independent pieces of state, each a writable signal with its initial value and type.

.update() takes the current value and returns the next — here toggling the category off if you click the one already active.

.set(0) resets the highlight. Both writes mark dependents stale; nothing recomputes until something reads.

readonly protects the binding (you cannot reassign the field), while the signal's own API controls value mutation. Idiomatic Angular signal style.

computed(): derived, lazy, memoized

The filtered result is a computed signal. It reads the three inputs and produces the visible rows — and it only does that work when someone actually reads it.

skills.component.ts 
readonly filtered = computed<Row[]>(() => {
  const q = this.query().trim().toLowerCase();
  const cat = this.activeCat();
  return this.rows.filter(
    (r) => (!cat || r.group === cat) && (!q || r.hay.includes(q)),
  );
});
What Angular does

Reading query() and activeCat() inside the body registers them as this computed's dependencies — automatically, no dependency list to declare.

this.rows is a plain array, not a signal, so it is not tracked — only the two signal reads are.

The result is cached. Read filtered() ten times with no writes in between and the filter runs once.

After query or activeCat changes, the cache is invalidated; the next read recomputes. If neither changed, the old array is returned by reference.

🕸️
Dependencies are dynamic

Only signals actually read on the last run count. If a computed short-circuits before reading a signal (say, an early return), that signal is temporarily not a dependency and its changes will not wake the computed. The graph re-forms on every evaluation.

Push notification, pull evaluation — and glitch-free

Writes only push a stale flag; values are pulled lazily on read. A consequence: within one synchronous update, a computed can never be observed in a half-updated ("glitched") state, because it recomputes at most once, on demand, after all writes have landed.

effect(): when you need a side effect, not a value

computed derives pure values. When a signal change must touch the outside world — the DOM, localStorage, logging — you use an effect. The theme service is the textbook case.

theme.service.ts 
readonly theme = signal<Theme>(this.initial());

constructor() {
  effect(() => {
    const t = this.theme();
    document.documentElement.setAttribute('data-theme', t);
    document.documentElement.style.colorScheme = t;
    localStorage.setItem('portfolio-theme', t);
  });
}
What Angular does

The effect reads theme(), so theme becomes its one dependency.

It runs once immediately after creation to sync the initial state, then again every time theme is written.

Each run writes the data-theme attribute (which flips every var(--*) token across the app), sets color-scheme, and persists the choice.

Created in the constructor, it inherits the service's injection context, so Angular auto-destroys it when the service is destroyed — no manual teardown.

🧭
Don't reach for effect to compute state

If you find an effect writing to another signal, you almost always wanted a computed. Effects are for the boundary between the reactive graph and the imperative world (DOM, storage, analytics). Overusing them recreates the tangled callback graphs that signals exist to eliminate.

Signals vs. RxJS: why this app chose signals

Both model change over time. They solve different shapes of problem, and this codebase is deliberately signals-only.

🎚️

Signals → state

Synchronous "what is the value now" questions. Always have a value, read like a variable, integrate directly with change detection. Ideal for UI state.

🌊

RxJS → streams

Asynchronous event pipelines — debouncing, cancellation, merging websockets. Powerful operators, but overkill for "what colour is the theme".

💡
The decision heuristic

This app has no network, no cancellation, no event coordination — only synchronous UI state. That is the exact niche signals own. RxJS would add ceremony (subscriptions, teardown, an async pipe) for zero benefit. Reach for RxJS when you have genuine async streams to orchestrate; reach for signals for everything that is "current value of X".

Signals are read inside services and injected into components. That injection — the inject() calls we keep glossing over — is next.

Dependency injection

How a component gets a service without ever calling new: injectors, tokens, the injector hierarchy, and the inject() function.

Request dependencies, never construct them

Dependency injection (DI) is the backbone that wires this app together. A component does not build its own ResumeService; it asks for one, and an injector supplies it.

1

Token
usually a class, e.g. ResumeService

→
2

Injector
looks up the token in its provider map

→
3

Instance
returns the cached one, or creates it

🗝️
What a "token" is

A DI token is the lookup key. A class is its own token, which is why inject(ResumeService) just works. For values without a class — a config object, a base URL — you create an InjectionToken to serve as the key.

Four dependencies, one line each

The root component pulls in everything it needs through inject() — the modern function form of DI.

app.component.ts 
export class AppComponent implements AfterViewInit {
  private themeSvc = inject(ThemeService);
  private sections = inject(SectionService);
  private scrollSpy = inject(ScrollSpyService);
  private resume = inject(ResumeService);

  readonly theme = this.themeSvc.theme;
  readonly nav = this.sections.sections;
What Angular does

Each inject(Token) asks the current injector for the instance registered under that token.

These run in an injection context — field initializers during component construction — which is the only place inject() is allowed.

Because all four services are app-wide singletons, every component that injects them shares the exact same instance.

The component then re-exposes service signals (theme, nav) as its own fields for the template to bind to — a thin pass-through.

providedIn: 'root' — singleton + tree-shaking

How does the injector know how to make a ResumeService? The service declares its own provider, right on the class.

resume.service.ts 
@Injectable({ providedIn: 'root' })
export class ResumeService {
  readonly profile = PROFILE;
  readonly links = LINKS;
  readonly projects = PROJECTS;
  readonly skills = SKILLS;
  // …the single source of truth, wrapped for DI
}
What Angular does

@Injectable({ providedIn: 'root' }) registers the class with the root injector as a lazily-created singleton.

"Lazily" matters: the instance is created the first time something injects it, not at boot.

Because registration lives on the class itself, the bundler can tree-shake the service away entirely if nothing injects it.

This service is a facade: it wraps the plain resume.ts data module so components depend on an injectable seam, not a hard import.

🧪
Why wrap static data in a service at all?

The data never changes at runtime, so why not import resume.ts directly? Because DI gives a substitution point. In a unit test you can provide a stub ResumeService with three fake projects; a direct import would be un-mockable. This is separation of concerns bought cheaply — the whole reason the codebase routes all content through ResumeService and SectionService.

Resolution walks upward

When you inject(X), Angular checks this element's injector, then its parent elements' injectors, then the environment injectors, up to root. First match wins. Miss everything and it throws NullInjectorError. This upward walk is why a service provided high in the tree is visible to everything below it.

Element-scoped injection: ElementRef

Not every dependency is a global service. The reveal directive injects its own host element — a token only the element injector can satisfy.

reveal.directive.ts 
export class RevealDirective implements OnInit, OnDestroy {
  @Input() revealDelay = 0;
  private el = inject(ElementRef<HTMLElement>);
  private observer?: IntersectionObserver;
What Angular does

inject(ElementRef) resolves against the directive's own element injector, returning a handle to the exact DOM node the directive is attached to.

This could never come from the root injector — it is inherently per-element. That is what the element-injector layer exists for.

The same inject() function transparently handles both global singletons and element-scoped tokens; the hierarchy decides where each resolves.

inject() vs. constructor injection

Both forms do the same DI lookup. The function form is what this codebase uses everywhere, and it has concrete advantages.

🧬

Composes cleanly

Shared setup can be extracted into plain functions that call inject(), reused across classes — impossible with constructor parameters.

🧾

Better types

No reliance on parameter decorators or emitDecoratorMetadata; generics like ElementRef<HTMLElement> flow through naturally.

🪜

No constructor noise

Dependencies read as field initializers instead of a long constructor signature; subclassing does not force super-call plumbing.

⏱️
The one rule: injection context

inject() only works while Angular is constructing the class — in field initializers or the constructor body. Call it later, inside a click handler or a setTimeout, and it throws. If you must inject after construction, capture an Injector during construction and use runInInjectionContext.

Injected services hold signals; templates read them. What actually decides when Angular re-reads and repaints? That is change detection — and the zone that drives it.

06

Change detection & zones

The loop that keeps the DOM in sync with your data — what triggers it, how Zone.js knows, how OnPush and signals prune it, and where lifecycle hooks fit.

The loop, in one sentence

Change detection (CD) is: something might have changed → walk the component tree top-to-bottom → for each binding, compare the new value to the last one → update only the DOM that differs. The open question is what that "something might have changed" trigger actually is.

🖱️
DOM events

click, input, keydown — anything with an event binding, plus events anywhere in the tree.

⏲️
Timers & async

setTimeout, setInterval, promises resolving, XMLHttpRequest/fetch completing.

📡
Signal writes

A signal read in a template changing value marks that specific view for check.

Zone.js: how Angular knows an event happened

Angular does not ask you to call "refresh". Instead Zone.js monkey-patches every async browser API at startup. When any patched task finishes, Angular's zone (NgZone) is notified and schedules a CD pass. That runtime is installed by one provider from Module 01.

app.config.ts 
export const appConfig: ApplicationConfig = {
  providers: [
    provideZoneChangeDetection({ eventCoalescing: true }),
    provideRouter(routes),
  ],
};
What Angular does

Installs the NgZone-based change detection scheduler.

eventCoalescing: true merges multiple events firing within the same task into a single CD run, instead of one run per event.

Without coalescing, a burst of events (say, a native control emitting input + change together) could trigger redundant back-to-back tree walks.

This is the one line that makes the whole "just mutate state and the view updates" experience work.

🔭
Escaping the zone

Because Zone.js patches everything, high-frequency work (scroll, requestAnimationFrame loops, third-party libs) can trigger CD far too often — "zone pollution." The escape hatch is NgZone.runOutsideAngular(), which runs code in a zone Angular ignores. Remember this; the scroll-spy at the end of this module is a hand-rolled version of the same idea.

Pruning the walk: OnPush & dirty marking

A full top-down walk on every tick is wasteful for large trees. OnPush lets a component say "only check me when you have a reason." The typewriter opts in.

typewriter.component.ts 
@Component({
  selector: 'app-typewriter',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<span class="tw">{{ text() }}</span>`,
})
What Angular does

This subtree is skipped during a normal tick unless it is marked dirty.

It gets marked when: an @Input reference changes, an event fires inside it, markForCheck() is called, or — crucially — a signal it reads changes.

{{ text() }} reads a signal, so each typewriter tick marks precisely this view dirty and nothing else on the page.

That is the modern pairing: OnPush + signals gives fine-grained updates without any manual markForCheck calls.

🧭
Dirty marks travel to the root

When a view is marked dirty, Angular flags a path of dirty flags from that component up to the root. On the next tick, CD walks from the top but skips any OnPush subtree not on a dirty path. So marking one deep signal repaints one view, yet the traversal still starts at the root — the flags just tell it where it is allowed to descend.

Lifecycle hooks: the fixed order

Between construction and destruction, Angular calls hooks in a guaranteed sequence. This app uses three of them. ngAfterViewInit is where the root wires up the scroll-spy — because only then does the view (and its section elements) exist.

1
constructor

DI resolves; inject() runs. ThemeService registers its effect here. No DOM yet.

2
ngOnInit

Inputs are set; safe to initialise. The reveal directive and typewriter start their observers/timers here.

3
ngAfterViewInit

The component's view and children exist. AppComponent starts the scroll-spy now — the section DOM nodes it measures are finally present.

4
ngOnDestroy

Teardown. Directives disconnect observers, the typewriter clears its timer, the scroll-spy removes listeners. No leaks.

app.component.ts 
ngAfterViewInit(): void {
  this.scrollSpy.init(this.sections.ids());
}
What Angular does

Runs once, after the full view tree is rendered.

It hands the ordered section ids to the scroll-spy, which begins measuring their positions.

Doing this in the constructor or ngOnInit would measure elements that do not exist yet — hence the specific hook.

A deliberate detour around the loop

The scroll-spy fires on every scroll event — potentially hundreds per second. Running full CD that often would be ruinous. Its design sidesteps the problem: throttle with requestAnimationFrame and only ever write a signal, letting the signal decide what repaints.

scroll-spy.service.ts 
private onScroll(): void {
  this.scrolled.set(window.scrollY > 20);
  if (this.ticking) return;
  this.ticking = true;
  requestAnimationFrame(() => {
    this.compute();
    this.ticking = false;
  });
}
What Angular does

Scroll fires constantly, but the ticking guard collapses a burst into at most one compute() per animation frame.

compute() figures out the active section and calls activeId.set(...) — the only escape into the reactive world.

The active-section highlight is a signal binding, so exactly the views reading activeId refresh — not the whole page.

Deterministic by design: the active section is derived from absolute scroll position each frame, so it never gets "stuck" the way an IntersectionObserver callback can.

🔮
Where this is all heading: zoneless

Angular is moving toward dropping Zone.js entirely. In a zoneless app (experimental in v19), signals become the primary CD trigger — no monkey-patching, smaller bundles, more predictable timing. Code already written the way this app writes it — state in signals, side effects in effects — is the code that ports to zoneless with the least friction.

The scroll-spy and reveal directive keep pointing at one last actor we have not opened up: directives themselves, and how they reach into the DOM.

07

Directives & the DOM

Attaching behaviour to any element without a template — host bindings, ElementRef, browser APIs, and disciplined teardown.

A directive is a component without a view

A component is a directive that happens to have a template. An attribute directive keeps the class, the decorator, the DI, and the lifecycle hooks — but renders nothing of its own. It decorates an element that is already there.

reveal.directive.ts 
@Directive({
  selector: '[appReveal]',
  standalone: true,
  host: { '[attr.data-reveal]': '""' },
})
export class RevealDirective implements OnInit, OnDestroy {
What Angular does

selector: '[appReveal]' — an attribute selector. Any element with appReveal on it gets an instance of this class attached.

standalone: true — like components, directives are standalone in v19; a template imports this directive by listing it in imports.

host writes a data-reveal attribute onto the host element declaratively — no imperative DOM call needed for the marker.

Used as <app-section-header appReveal /> or <div appReveal [revealDelay]="80"> throughout the sections.

🧷
Two kinds of directive

This is an attribute directive — it changes an existing element. The other kind, structural directives (*ngIf, *ngFor), added or removed elements. In v19 the structural role is handled by built-in control flow (Module 03), so most directives you write today are attribute directives like this one.

Reaching the real DOM node

To observe when an element scrolls into view, the directive needs the actual DOM node. It injects ElementRef (Module 05) and drives a browser API directly.

reveal.directive.ts — ngOnInit 
ngOnInit(): void {
  const node = this.el.nativeElement as HTMLElement;
  const reduce = matchMedia('(prefers-reduced-motion: reduce)').matches;
  if (reduce || typeof IntersectionObserver === 'undefined') {
    node.classList.add('is-visible');
    return;
  }
  this.observer = new IntersectionObserver((entries) => {
    for (const e of entries)
      if (e.isIntersecting) {
        node.classList.add('is-visible');
        this.observer?.unobserve(node);
      }
  }, { threshold: 0.12, rootMargin: '0px 0px -8% 0px' });
  this.observer.observe(node);
}
What Angular does

nativeElement is the raw DOM node; the directive manipulates it directly.

Accessibility first: if the user prefers reduced motion, skip the animation entirely and just reveal the element. A graceful degrade also covers environments without IntersectionObserver.

An IntersectionObserver watches the node; when it enters the viewport it adds is-visible (which triggers the CSS transition) and immediately unobserves — a one-shot reveal.

This runs entirely outside Angular's bindings — pure DOM work — which is exactly why it is cheap.

🛠️
Direct DOM access vs. Renderer2

Touching nativeElement directly is fine here because the work is trivial and browser-only. For anything that must run in non-DOM environments (server-side rendering, web workers) or that you want abstracted, Angular offers Renderer2. Knowing both exist lets you pick deliberately rather than by habit.

Inputs configure the directive

The directive is parameterised the same way a component is — with an @Input. Here it staggers reveal animations.

reveal.directive.ts + usage 
@Input() revealDelay = 0;
// …
if (this.revealDelay)
  node.style.transitionDelay = `${this.revealDelay}ms`;

// in a section template:
<div class="cmd" appReveal [revealDelay]="80">
What Angular does

@Input() revealDelay exposes a bindable property with a default of 0.

[revealDelay]="80" in a host template passes a value in — the same property-binding syntax from Module 03, targeting a directive input instead of a DOM property.

The directive uses it to offset the CSS transition, producing staggered reveals as a section scrolls in.

Teardown is not optional

Anything a directive or service subscribes to — observers, global listeners, timers — must be released in ngOnDestroy, or it leaks. Both long-lived pieces in this app clean up after themselves.

reveal.directive.ts & scroll-spy.service.ts 
// RevealDirective
ngOnDestroy(): void {
  this.observer?.disconnect();
}

// ScrollSpyService
ngOnDestroy(): void {
  window.removeEventListener('scroll', this.handler);
  window.removeEventListener('resize', this.handler);
}
What Angular does

When the directive's element is removed, Angular calls ngOnDestroy; disconnect() stops the observer and frees the node reference.

The service stores its listener as a stable handler field precisely so it can remove the same function reference later.

Skip either and you get a memory leak: dead observers and listeners piling up across navigations.

Note the effect in ThemeService needs none of this — effects auto-destroy with their injection context.

🧹
The teardown mental checklist

For every addEventListener, new IntersectionObserver, setInterval, or manual RxJS subscribe you create, ask "who calls the matching remove/disconnect/clear/unsubscribe?" If the answer is "nobody", it belongs in ngOnDestroy — or use takeUntilDestroyed() / DestroyRef to automate it.

The capstone component

The "Query the résumé" section is a fake database console: tabs pick a saved query, a Run button fakes execution, and results animate in. Nothing is real — no SQL engine, no database. That makes it a clean specimen for watching the framework work, and it touches Modules 02–07 at once.

🎨
A pure function does the highlighting

highlightSql tokenizes each query string once — no reactivity involved.

📡
Signals hold the interaction state

active, running, ms — plus a computed that derives the current query.

🔁
Lifecycle drives the side effects

An IntersectionObserver auto-runs the first query on scroll-in; teardown cleans it up.

The tokenizer is deliberately not reactive

Syntax highlighting is a pure transformation: same SQL string in, same token array out, forever. So it is a plain function called once at construction — not a signal, not a computed. This is the "don't over-signal" lesson from Module 04 in practice.

sql-highlight.ts 
export function highlightSql(sql: string): SqlToken[][] {
  return sql.split('\n').map((line) => {
    const out: SqlToken[] = [];
    let m: RegExpExecArray | null;
    TOKEN.lastIndex = 0;
    while ((m = TOKEN.exec(line))) {
      if (m[1]) out.push({ t: m[1], c: 'tok-str' });
      else if (m[2]) {
        const w = m[2].toLowerCase();
        out.push({ t: m[2], c: KEYWORDS.has(w) ? 'tok-key' : FUNCTIONS.has(w) ? 'tok-fn' : 'tok-id' });
      } else if (m[3]) out.push({ t: m[3], c: 'tok-num' });
      else if (m[4]) out.push({ t: m[4], c: '' });
      else out.push({ t: m[5], c: 'tok-op' });
    }
    return out;
  });
}
What it does

Split the SQL into lines, then walk each line with one regex that alternates over string / word / number / whitespace / operator.

A matched word is cross-checked against KEYWORDS and FUNCTIONS sets to tag it tok-key, tok-fn, or plain tok-id.

Each token carries the text t and a CSS class c — the [class]="t.c" the template binds to.

Called eagerly in the component's queries field initializer. The result never changes, so caching it in a signal would add machinery for nothing.

🧮
Reactivity is for change, not for computation

A common beginner reflex is to wrap everything in signals. The discipline here: values that never change after creation stay as plain data; only genuinely mutating state (which tab is active, is it running) becomes a signal. Fewer reactive nodes means a smaller graph and less to reason about.

Three signals and a derived query

All the live state is three writable signals plus one computed that selects the active query object — a direct echo of the skills filter from Module 04.

query-console.component.ts 
readonly active = signal(0);
readonly running = signal(false);
readonly ms = signal('0.6');
readonly q = computed(() => this.queries[this.active()]);

select(i: number): void {
  if (i === this.active()) { this.run(); return; }
  this.active.set(i);
  this.run();
}
What Angular does

active holds the selected tab index; running gates the loading state; ms is the fake execution time shown in the footer.

q = computed(...) derives the whole current query object from active. Every template reference to q() — editor, columns, rows, footer — re-points in one step when the tab changes.

select(i) writes active then triggers a run. Clicking the already-active tab re-runs instead of no-oping — a small UX choice made trivial by signals.

One write to active invalidates q, which the template pulls lazily on the next change-detection pass (Module 06).

Real @if / @else in the template

Module 03 taught @switch with an illustrative snippet; here is the console's genuine conditional — the loading state versus the results grid, driven by the running signal.

query-console.component.ts — template 
@if (running()) {
  <div class="running mono">
    <span class="spin"></span> executing on career.db…
  </div>
} @else {
  <div class="grid"> <!-- results table --> </div>
}
What Angular does

@if (running()) reads the signal directly in the template — no *ngIf, no async pipe.

While running is true, only the spinner subtree exists in the DOM; the results subtree is not rendered at all.

@else swaps to the grid when execution "finishes". Angular creates and destroys the two subtrees as the boolean flips.

Because it reads a signal, the swap is precise: flipping running marks exactly this view for check.

Lifecycle: auto-run on scroll, then clean up

The console runs its first query automatically when it scrolls into view — a side effect wired in ngAfterViewInit and torn down in ngOnDestroy, exactly the discipline from Modules 06 and 07.

query-console.component.ts 
ngAfterViewInit(): void {
  this.observer = new IntersectionObserver((entries) => {
    for (const e of entries)
      if (e.isIntersecting && !this.hasAutoRun) {
        this.hasAutoRun = true;
        this.run();
        this.observer?.disconnect();
      }
  }, { threshold: 0.35 });
  this.observer.observe(this.host.nativeElement);
}

ngOnDestroy(): void {
  clearTimeout(this.timer);
  this.observer?.disconnect();
}
What Angular does

In ngAfterViewInit the host element exists, so it can be observed. When it enters the viewport, run once and immediately disconnect — the hasAutoRun guard plus disconnect make it strictly one-shot.

run() sets running(true) and a setTimeout that flips it back after ~430ms, faking execution latency.

ngOnDestroy clears the pending timer and disconnects the observer — no dangling callbacks if the component is torn down mid-run.

A reduced-motion check (elsewhere in run()) skips the spinner entirely — the accessibility floor again.

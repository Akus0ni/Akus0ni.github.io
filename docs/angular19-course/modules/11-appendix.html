One feature that exercises every system in the course: a pure tokenizer, signals, a computed, real @if/@else, and lifecycle-driven side effects.

The capstone component

The "Query the résumé" section is a fake database console: tabs pick a saved query, a Run button fakes execution, and results animate in. Nothing is real — no SQL engine, no database. That makes it a clean specimen for watching the framework work, and it touches Modules 02–07 at once.

📡

🔁

The tokenizer is deliberately not reactive

Syntax highlighting is a pure transformation: same SQL string in, same token array out, forever. So it is a plain function called once at construction — not a signal, not a computed. This is the "don't over-signal" lesson from Module 04 in practice.

sql-highlight.ts 
export function highlightSql(sql: string): SqlToken[][] {
  return sql.split('\n').map((line) => {
    const out: SqlToken[] = [];
    let m: RegExpExecArray | null;
    TOKEN.lastIndex = 0;
    while ((m = TOKEN.exec(line))) {
      if (m[1]) out.push({ t: m[1], c: 'tok-str' });
      else if (m[2]) {
        const w = m[2].toLowerCase();
        out.push({ t: m[2], c: KEYWORDS.has(w) ? 'tok-key' : FUNCTIONS.has(w) ? 'tok-fn' : 'tok-id' });
      } else if (m[3]) out.push({ t: m[3], c: 'tok-num' });
      else if (m[4]) out.push({ t: m[4], c: '' });
      else out.push({ t: m[5], c: 'tok-op' });
    }
    return out;
  });
}
What it does

Split the SQL into lines, then walk each line with one regex that alternates over string / word / number / whitespace / operator.

A matched word is cross-checked against KEYWORDS and FUNCTIONS sets to tag it tok-key, tok-fn, or plain tok-id.

Each token carries the text t and a CSS class c — the [class]="t.c" the template binds to.

Called eagerly in the component's queries field initializer. The result never changes, so caching it in a signal would add machinery for nothing.

🧮
Reactivity is for change, not for computation

A common beginner reflex is to wrap everything in signals. The discipline here: values that never change after creation stay as plain data; only genuinely mutating state (which tab is active, is it running) becomes a signal. Fewer reactive nodes means a smaller graph and less to reason about.

Three signals and a derived query

All the live state is three writable signals plus one computed that selects the active query object — a direct echo of the skills filter from Module 04.

query-console.component.ts 
readonly active = signal(0);
readonly running = signal(false);
readonly ms = signal('0.6');
readonly q = computed(() => this.queries[this.active()]);

select(i: number): void {
  if (i === this.active()) { this.run(); return; }
  this.active.set(i);
  this.run();
}
What Angular does

active holds the selected tab index; running gates the loading state; ms is the fake execution time shown in the footer.

q = computed(...) derives the whole current query object from active. Every template reference to q() — editor, columns, rows, footer — re-points in one step when the tab changes.

select(i) writes active then triggers a run. Clicking the already-active tab re-runs instead of no-oping — a small UX choice made trivial by signals.

One write to active invalidates q, which the template pulls lazily on the next change-detection pass (Module 06).

Real @if / @else in the template

Module 03 taught @switch with an illustrative snippet; here is the console's genuine conditional — the loading state versus the results grid, driven by the running signal.

query-console.component.ts — template 
@if (running()) {
  <div class="running mono">
    <span class="spin"></span> executing on career.db…
  </div>
} @else {
  <div class="grid"> <!-- results table --> </div>
}
What Angular does

@if (running()) reads the signal directly in the template — no *ngIf, no async pipe.

While running is true, only the spinner subtree exists in the DOM; the results subtree is not rendered at all.

@else swaps to the grid when execution "finishes". Angular creates and destroys the two subtrees as the boolean flips.

Because it reads a signal, the swap is precise: flipping running marks exactly this view for check.

Lifecycle: auto-run on scroll, then clean up

The console runs its first query automatically when it scrolls into view — a side effect wired in ngAfterViewInit and torn down in ngOnDestroy, exactly the discipline from Modules 06 and 07.

query-console.component.ts 
ngAfterViewInit(): void {
  this.observer = new IntersectionObserver((entries) => {
    for (const e of entries)
      if (e.isIntersecting && !this.hasAutoRun) {
        this.hasAutoRun = true;
        this.run();
        this.observer?.disconnect();
      }
  }, { threshold: 0.35 });
  this.observer.observe(this.host.nativeElement);
}

ngOnDestroy(): void {
  clearTimeout(this.timer);
  this.observer?.disconnect();
}
What Angular does

In ngAfterViewInit the host element exists, so it can be observed. When it enters the viewport, run once and immediately disconnect — the hasAutoRun guard plus disconnect make it strictly one-shot.

run() sets running(true) and a setTimeout that flips it back after ~430ms, faking execution latency.

ngOnDestroy clears the pending timer and disconnects the observer — no dangling callbacks if the component is torn down mid-run.

A reduced-motion check (elsewhere in run()) skips the spinner entirely — the accessibility floor again.

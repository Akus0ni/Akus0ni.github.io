Templates & built-in control flow

How binding syntax connects class state to the DOM, and how @for / @if / @switch replaced the old structural directives with compiled control flow.

Five ways to bind, one direction each

Every character of dynamism in an Angular template is one of five binding forms. Each compiles to a specific Ivy instruction; there is no general expression evaluator at runtime.

{{ }}

Interpolation

{{ expr }} — writes a string into a text node. Class → DOM.

[ ]

Property binding

[prop]="expr" — sets a DOM property or component input. Class → DOM.

( )

Event binding

(event)="handler()" — runs a method on a DOM/output event. DOM → class.

[( )]

Two-way

[(x)] — sugar for a property binding + an xChange event. Both directions.

[attr.]

Attribute / class / style

[attr.x], [class.y], [style.z] — bind to attributes, classes, styles when no DOM property exists.

🧩
Property vs attribute

[value] sets the live DOM property; [attr.aria-selected] sets the HTML attribute. Angular exposes both because ARIA state, data-*, and SVG live in attribute-space with no matching JS property. This app uses [attr.data-idx] and [attr.aria-selected] for exactly that reason.

Four binding forms in one input

The skills search box is a compact tour of the syntax — property, event, interpolation, and a two-way pattern split into its parts.

skills.component.ts — template 
<input #box class="cmd-input mono" type="text"
       [ngModel]="query()" (ngModelChange)="onQuery($event)"
       (keydown)="onKey($event)"
       placeholder="Filter {{ rows.length }} skills…" />
<span class="cmd-count mono">{{ filtered().length }}</span>
What Angular does

[ngModel]="query()" — property binding pushing the query signal's value into the input. Instead of the usual [(ngModel)] two-way sugar, it is split manually.

(ngModelChange)="onQuery($event)" — the other half: on each keystroke, call onQuery with the new value, which writes the signal. Splitting the two-way binding lets the handler also reset the selection index.

(keydown)="onKey($event)" — event binding for arrow-key navigation; $event is the raw KeyboardEvent.

{{ rows.length }} and {{ filtered().length }} — interpolation, the second reading a signal so the count re-renders as results filter.

🔁
[(x)] is just [x] + (xChange)

Two-way binding is not magic — it is a naming convention. [(ngModel)]="q" desugars to [ngModel]="q" (ngModelChange)="q = $event". Seeing it split here shows the seam: any two-way binding can be pulled apart when you need to intercept the update, exactly as this code does to reset the highlighted row.

@for and the mandatory track

Since v17, loops and conditionals are built-in blocks in the template — not directives. Here is the results list.

skills.component.ts — @for 
@for (r of filtered(); track r.name; let i = $index) {
  <li role="option" [attr.aria-selected]="i === index()">
    <div class="cmd-row" [class.sel]="i === index()"
         (mouseenter)="index.set(i)"> … </div>
  </li>
} @empty {
  <li class="cmd-none mono">// no symbols match</li>
}
What Angular does

Iterate filtered() — itself a signal, so the list re-renders whenever the filter recomputes.

track r.name is required. It gives each row a stable identity so Angular can match old DOM to new data across re-renders.

let i = $index aliases the built-in index variable; also available: $count, $first, $last, $even, $odd.

@empty renders when the collection is empty — replacing the old pattern of a sibling *ngIf guard.

Why track decides reuse vs. rebuild

When the array changes, Angular diffs the new items against the tracked keys. Matching keys → the existing DOM node and its component state are reused, only bindings update. Missing keys → nodes are destroyed; new keys → nodes are created. Step through a filter operation:

The classic track mistake

Tracking by object identity (track item) when the array is rebuilt from fresh objects — e.g. after a map or an HTTP response — makes every key look new, so Angular destroys and recreates every row on each change. That is the single most common Angular list-performance bug. Track by a stable domain id (r.name here), or by $index only for static lists.

@switch, and when not to reach for it

The third control-flow block, @switch, picks one branch by strict equality — no fall-through, no break. This codebase never uses it, so the snippet below is illustrative: imagine the SQL console wanting different markup per token kind.

illustrative — not in the repo 
@switch (token.c) {
  @case ('tok-key') { <strong class="kw">{{ token.t }}</strong> }
  @case ('tok-str') { <em class="str">{{ token.t }}</em> }
  @case ('tok-fn')  { <a class="fn">{{ token.t }}</a> }
  @default          { <span>{{ token.t }}</span> }
}
What Angular does

Evaluate token.c once, then render the single @case whose value === it.

No fall-through: exactly one branch renders, so there is no break and no accidental double-render.

@default covers unmatched values — here, plain identifiers and operators.

Each branch is a full template region, so the cases can differ in structure, not just attributes.

🎯
Why the real console does not use @switch

Every token renders the same element and only the class differs. So the console writes one line — <span [class]="t.c">{{ t.t }}</span> — a plain class binding, not seven @case branches. Rule of thumb: reach for @switch when the branches produce different markup or components; reach for [class]/[ngClass] when only styling varies. Choosing the lighter tool is the mark of reading templates well.

Blocks, not directives

Old Angular used structural directives — *ngIf, *ngFor, *ngSwitch — imported from CommonModule. The new control flow is part of the template language itself.

⚡
Nothing to import

@if/@for/@switch are compiler built-ins. No CommonModule dependency, no directive resolution step.

🏎️
Compiles to direct instructions

The block lowers straight to conditional/loop instruction calls — no ng-template indirection, less overhead than the directive it replaces.

✅
track is enforced

What used to be an optional trackBy function is now a required part of the syntax — the framework makes the fast path the default path.

Templates read state; the next question is what that state actually is. In this codebase it is signals — the reactive core.

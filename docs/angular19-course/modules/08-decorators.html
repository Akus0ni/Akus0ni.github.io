Decorators, in depth

The @ you have seen on every class so far is just a function that staples metadata onto that class. Here is the full taxonomy — and what the compiler does with it before your code ever runs.

A decorator is a function, not magic

Think of a class as a suitcase and a decorator as the luggage tag the check-in desk staples to the handle. The suitcase is unchanged — same contents, same zips — but the tag carries routing information the baggage system reads later. A decorator works the same way: it attaches configuration to a class without altering the class body.

🧳
The class

Plain TypeScript — fields, a constructor, methods. It has no idea it is a component.

🏷️
The decorator

@Component({…}) records "this class is a rendering unit, here is its template" as metadata.

🛄
The compiler

Reads the tag and routes the class into the framework — no runtime guesswork.

The tag, on a real class

The reveal directive in this codebase is nothing but a class with a tag. Strip the @Directive line and you have inert TypeScript; add it back and Angular knows how to wire it.

reveal.directive.ts 
@Directive({
  selector: '[appReveal]',
  standalone: true,
})
export class RevealDirective implements OnInit, OnDestroy {
  // …plain class members below
}
What the decorator records

@Directive(config) is a function call. It runs when the file is defined and stamps the config object onto the class.

The selector tells the compiler: wherever an element has the appReveal attribute, attach an instance of this class to it.

The class itself never references Angular. All the "this is a directive" knowledge lives in the tag, not the body.

Remove the decorator and the class still compiles as TypeScript — it just stops being anything Angular can see.

The tag is read at build time, then thrown away

This is the part most tutorials skip. With Ahead-of-Time compilation — the v19 default — the Angular compiler reads your decorators during the build and rewrites each class to carry a static definition field. At runtime, the framework reads that field, not the decorator.

Why this matters when you steer a build

Because decorators are compiled away, most decorator mistakes are caught at build time, not as a mysterious runtime crash. And because Angular's own decorators are read statically, this app does not ship the reflect-metadata polyfill — smaller bundle, one less thing to configure.

Three places a decorator can sit

Every Angular decorator lives in exactly one of three positions on a class. Knowing which is which tells you instantly what a symbol does.

🏛️

Class decorators

Sit above the class keyword and declare what kind of thing it is: @Component, @Directive, @Injectable, @Pipe, @NgModule.

🔌

Member decorators

Sit on a field or method to expose it: @Input, @Output, @HostBinding, @HostListener, @ViewChild, @ContentChild.

🎚️

Parameter decorators

Sit on a constructor parameter to modify a DI request: @Inject, @Optional, @Self, @SkipSelf, @Host.

🗺️
A quick read-the-position heuristic

Above the class → it defines a building block. On a property → it connects that property to the outside world. On a constructor argument → it fine-tunes where an injected dependency comes from. That is the entire map.

The four class decorators, side by side

Each class decorator generates a different static definition and slots the class into a different part of the framework.

@Component A directive with a template. Generates ɵcmp; renders a view. Seen on every file in sections/ and components/.
@Directive Behaviour attached to an existing element, no template. Generates ɵdir. This app's RevealDirective.
@Injectable A class that can be provided and injected. Generates ɵinj/ɵfac. All four core/ services carry it.
@Pipe A reusable value transform usable in templates. Generates ɵpipe. Covered in Module 10.
🔎
@Component is @Directive plus a view

Internally a component is a directive that also owns a template — which is exactly why Module 07 could call a directive "a component without a view." Same machinery, one extra capability.

Member decorators: @Input up close

This codebase uses one member decorator heavily — @Input — to let a parent pass data into a child. Here it is on the typewriter component, verbatim.

typewriter.component.ts 
export class TypewriterComponent {
  @Input({ required: true }) phrases: string[] = [];
  // …animation state below
}
What the decorator wires

@Input marks phrases as a public entry point: a parent template can write [phrases]="myList" and the value lands here.

The { required: true } option makes the compiler error if a parent forgets to pass it — a build-time guarantee, not a runtime surprise.

Other options exist: alias renames the binding, and transform runs a function on the value as it arrives.

Without the decorator, phrases would be an ordinary private-ish field the parent could never reach.

🧩
The rest of the member family

Angular ships more member decorators this app happens not to need: @Output emits events to the parent, @HostListener reacts to DOM events on the host element, @HostBinding writes to a host attribute, and @ViewChild/@ContentChild grab references to elements. Module 09 shows their modern, signal-based replacements.

Parameter decorators: bending a DI lookup

Module 05 traced how inject() walks up the injector tree until it finds a provider. Parameter decorators (and their inject() option equivalents) let you change that walk when you need to.

🤷

@Optional

"If nobody provides this, give me null instead of throwing." Turns a hard dependency into a nice-to-have.

🚫

@Self

"Only look in this element's own injector. Do not walk up." Fails fast if the provider isn't local.

⏭️

@SkipSelf

"Skip my own injector; start the search at my parent." Useful for recursive components that must reach an ancestor.

🧱

@Host

"Walk up, but stop at my host component's boundary." Keeps a lookup from escaping into the wider app.

🎯
These are the knobs on the resolution walk

Everything Module 05 taught about "first match wins, walking upward" still holds — these decorators just change where the walk starts and what happens on a miss. In the inject() function form you pass the same intent as options: inject(Token, { optional: true, self: true }).

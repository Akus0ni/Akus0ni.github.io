Pipes

A declarative way to reshape a value for display only — right inside the template, without touching the underlying data or writing a method. Plus the pure/impure distinction that decides whether a pipe is cheap or costly.

A filter over the lens, not the subject

A pipe is like a photo filter: the original image on your camera roll is untouched, but what you see is warmer, cropped, brightened. A pipe transforms how a value appears in the template while the data behind it stays exactly as it was.

template syntax 
<!-- value | pipeName : argument -->
{{ startDate | date : 'MMM yyyy' }}
{{ name | uppercase }}
{{ stream$ | async }}
How to read it

The | sends the value on its left through the pipe named on its right.

Anything after a : is an argument — here, the date format string.

These three (date, uppercase, async) are built in. This app pulls them in through CommonModule, even though its templates don't currently use them.

The stored startDate never changes — only its rendered text does.

This codebase formats without a pipe — on purpose

The résumé needs to show "3.7 years" for a role. It does that with a plain function, not a pipe. Seeing the alternative sharpens what a pipe actually buys you.

utils/duration.ts 
/** Whole-year decimal between two 'YYYY-MM' dates. */
export function yearsBetween(from: string, to: string): string {
  const [fy, fm] = from.split('-').map(Number);
  const [ty, tm] = to.split('-').map(Number);
  return (((ty * 12 + tm) - (fy * 12 + fm)) / 12).toFixed(1);
}
What Angular does — and doesn't

A component calls yearsBetween(a, b) and stores the string. The template just prints it.

This is fine here because the value is computed once from static data that never changes at runtime.

A pipe earns its keep when the same formatting is reused across many templates, or must re-run as data changes — neither is true for this one-off calculation.

Knowing when not to reach for a pipe is as useful as knowing how to write one.

Turning that function into a pipe

If this formatting were needed in ten templates, you would promote it to a pipe: a class with the @Pipe decorator and a single transform method.

years.pipe.ts (illustrative) 
@Pipe({ name: 'years', standalone: true })
export class YearsPipe implements PipeTransform {
  transform(from: string, to: string): string {
    return yearsBetween(from, to);
  }
}
// in a template: {{ role.start | years : role.end }}
What each piece is

@Pipe({ name: 'years' }) registers the template name — the word after |.

PipeTransform forces the class to provide one method: transform.

The value before | becomes the first argument; each : arg becomes the next. So role.start is from, role.end is to.

standalone: true means a component just adds YearsPipe to its imports — no NgModule, same rule as components.

Pure vs impure — the setting that decides cost

This is the one pipe concept with real performance stakes. By default pipes are pure: Angular only re-runs transform when the input reference changes. An impure pipe re-runs on every single change-detection pass.

🧊

Pure (default)

Recomputes only when the input value or object reference changes. Cheap — memoized by design. Use this almost always.

🔥

Impure (opt-in)

{ pure: false }. Runs on every change-detection tick — fine for async, dangerous for heavy work. Reach for it rarely.

⚠️
The classic pure-pipe surprise

Because a pure pipe only re-runs when the reference changes, mutating an array in place (arr.push(x)) will not update a pipe that filters it — the reference is the same. The fix is to replace the array (arr = [...arr, x]), which is exactly the immutable style signals encourage. Recognising this saves hours of "why isn't my list updating?"

The takeaway

The pipe is correct. The caller mutates the array in place, so its reference never changes and the pure pipe never re-runs. Replace the array instead of pushing into it — the same immutable habit that keeps signals and OnPush happy.

The async pipe: the one you'll use most

The built-in async pipe is the standout impure pipe. It subscribes to an Observable or Promise, hands the latest value to the template, and — crucially — unsubscribes for you when the component is destroyed.

🔌
Subscribes automatically

{{ data$ | async }} — no manual .subscribe() in your component code.

🧹
Unsubscribes on destroy

It cleans up when the view goes away — sidestepping the memory-leak teardown lesson from Module 07.

📡
Signals reduce the need

In a signal-first app like this one, reading a signal in the template covers most of what async did — so you'll see it less here than in RxJS-heavy apps.

Signal inputs, outputs & queries

Angular 19's function-based replacements for @Input, @Output and @ViewChild. Same jobs — but the values are now signals, so they plug straight into the reactive graph from Module 04.

From mailbox to live feed

A @Input is like a mailbox: a value arrives, and if you want to react you have to be told the mail came (via ngOnChanges or a setter) and go check. A signal input() is a live feed: the value is a signal, so anything that reads it re-runs automatically when it changes. No notification step.

📬
@Input — the decorator way

A plain field. To react to changes you add ngOnChanges or a setter. It does not talk to signals on its own.

📡
input() — the signal way

Returns a read-only signal. Read it with this.value(); it slots directly into computed() and effect().

🧭
Decorators aren't gone — just for I/O

The class is still a component declared with the @Component decorator. What changes is how inputs, outputs and element queries are declared: functions called as field initializers, not member decorators. The Angular team recommends these for new code.

The same input, both ways

The typewriter component in this app uses the classic decorator. Here is that real line, next to the v19 signal form it maps to.

today — typewriter.component.ts 
// Decorator form (what this repo ships)
@Input({ required: true }) phrases: string[] = [];

// Signal form (the v19 equivalent)
phrases = input.required<string[]>();
What changes

The parent binding is identical either way: [phrases]="myList". Callers don't notice the switch.

input.required<T>() is the required variant — no default value, and the compiler enforces that a parent passes one.

The big difference is reading it: this.phrases() with parentheses, because it is now a signal, not a plain field.

Because it is a signal, a computed() that reads it recalculates the moment the parent changes the value — no ngOnChanges needed.

The three input flavours

One function, three shapes — chosen by whether the input is optional, mandatory, or needs a default plus a transform.

input(0) Optional input with a default value of 0. If the parent binds nothing, the signal reads the default.
input.required<T>() Mandatory. No default; the build fails if a parent forgets the binding — the signal equivalent of { required: true }.
input(0, { transform: … }) Runs a function on each incoming value — e.g. coerce a string attribute to a number before it lands.
🔒
Inputs are read-only by design

input() returns an InputSignal you can read but not .set(). That enforces one-way data flow: the parent owns the value, the child only consumes it. If a child needs to write back, that is a different tool — model(), two screens down.

output(): sending events upward

Where input() lets data flow down, output() lets a child announce that something happened. It replaces @Output + EventEmitter with one function.

a hypothetical filter chip 
// Old: decorator + EventEmitter
@Output() selected = new EventEmitter<string>();

// New: the output() function
selected = output<string>();

// Firing it looks the same either way
this.selected.emit('angular');
What changes

The parent still listens the same way: (selected)="onSelect($event)".

output() gives you back something with an emit() method — no new EventEmitter() boilerplate.

It is not a signal (an event is a moment, not a value), but it lives in the same function-based family and cleans up automatically.

model(): two-way in one line

Sometimes a child both reads a value and writes it back — a toggle, a search box. model() is a writable signal that a parent can bind two ways with the "banana in a box" syntax.

Queries: grabbing elements as signals

To reach a child element or component, @ViewChild used to hand you a value that was only reliable after ngAfterViewInit. The signal query viewChild() hands you a signal instead — read it anywhere, and it is defined when the element exists.

viewChild('ref') A signal to a single element/component in this component's own template. Replaces @ViewChild.
viewChildren('ref') A signal to a list of matching elements. Replaces @ViewChildren.
contentChild('ref') A signal to projected content passed in via <ng-content>. Replaces @ContentChild.
⏱️
Why "as a signal" fixes a classic bug

The old @ViewChild was undefined until the view initialised, so reading it too early silently returned nothing — a top-five Angular gotcha. A query signal reads consistently and lets a computed() or effect() wait for the element to appear, instead of you guessing the right lifecycle hook.

The decorator → signal cheat sheet

The whole migration, at a glance. Same intent on the left and right — the right is just reactive by default.

📥

@Input → input()

Data in. Now a read-only signal you call with ().

📤

@Output → output()

Events out. No more new EventEmitter().

🔁

@Input + @Output → model()

Two-way. One writable signal, banana-in-a-box binding.

🔎

@ViewChild → viewChild()

Element refs. Now a signal that's safe to read early.
